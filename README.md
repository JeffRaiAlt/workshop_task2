# Entity Resolution for Customer Profile Deduplication

Baseline-проект для поиска и объединения дубликатов клиентских профилей в базе маркетплейса по неполным и зашумлённым данным: email, телефон, ФИО, дата рождения, пол, география и поведенческие признаки. Проект направлен на повышение качества клиентской базы, точности персонализации и корректности аналитики. 

## Задача

В клиентской базе один и тот же пользователь может быть представлен несколькими профилями. Это приводит к искажению LTV, retention, RFM, ухудшению рекомендаций и ошибкам в loyalty-механиках. Текущая цель проекта — построить воспроизводимый pipeline для pairwise matching и последующего объединения профилей в единые сущности. 

## Текущее состояние

Сейчас реализован baseline-прототип:
- проведён EDA и анализ структуры датасета;
- подготовлен агрегированный профильный слой;
- реализована генерация пар профилей;
- обучена базовая pairwise-модель на CatBoost. 
В текущем датасете используется около **68k записей**, **61.9k уникальных профилей** и **53.4k сущностей**; также выделены **2654 multiprofile entity**, что подтверждает наличие реальной задачи дедупликации.  

## Что уже сделано

- Распаковка и агрегация `nonprocessingfeatures`, `realtimefeatures`, `fsfeatures`. 
- Формирование profile-level таблицы признаков.  
- Построение графового представления профилей и экспорт `nodes.parquet` / `edges.parquet`.
- Разбиение train / valid / test по `entityid` для более корректной валидации.  
- Обучение baseline CatBoost-классификатора для pairwise matching.  

## Текущие результаты

Для baseline-модели получены:
- `Valid ROC-AUC`: **0.7344**
- `Test ROC-AUC`: **0.7159**  

При стандартном пороге 0.5 модель пока не даёт пригодного качества по положительному классу, поэтому текущая версия рассматривается как исследовательский baseline, а не готовое решение для auto-merge. 

## Ограничения

Текущая версия проекта пока **не включает**:
- полноценный blocking / candidate generation;
- fuzzy/string similarity по текстовым полям;
- кластеризацию совпавших профилей в entity-компоненты;
- cluster-level evaluation;
- подбор бизнес-порогов для auto-merge и manual review;
- demo-интерфейс. 

Кроме того, в baseline рассматривается полный граф пар, а для **61 927** профилей это даёт ожидаемо около **1.92 млрд пар**, поэтому следующий обязательный шаг — переход к масштабируемому blocking.
