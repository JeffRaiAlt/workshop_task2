# 01. Первичный анализ датасета

Цель ноутбука: быстро понять масштаб данных, гранулярность, долю дублей и пригодность исходных полей для задачи Entity Resolution.

## Исходный датасет

| Показатель | Значение |
|---|---:|
| Строк / событий | 68 036 |
| Уникальных `profile_id` | 61 927 |
| Уникальных `entity_id` | 53 369 |
| Период данных | 2025-11-01 - 2026-05-11 |
| Всего возможных пар профилей | 1 917 445 701 |
| Реальных пар профилей одного клиента | 16 363 |

## Дубли в данных

| Показатель | Значение |
|---|---:|
| Клиентов с одним профилем | 45 296 |
| Доля клиентов с одним профилем | 84.87% |
| Клиентов с несколькими профилями | 8 073 |
| Доля клиентов с несколькими профилями | 15.13% |
| Профилей у клиентов с несколькими профилями | 16 631 |
| Доля таких профилей | 26.86% |
| Дополнительных профилей сверх первого | 8 558 |
| Доля дополнительных профилей | 13.82% |

## Заполненность identity-полей

| Поле | Покрытие профилей |
|---|---:|
| `email` | 100.00% |
| `sex` | 99.92% |
| `first_name` | 27.52% |
| `phone` | 7.52% |
| `last_name` | 2.40% |
| `birthday` | 0.60% |

## Ключевые выводы

1. Задача сильно несбалансирована: из почти 1.92 млрд возможных пар профилей только 16.4 тыс. являются реальными парами одного клиента.

2. Проверена базовая консистентность разметки: один `profile_id` не должен ссылаться на несколько разных `entity_id`. Это важно, потому что дальше `profile_id` используется как атомарная единица сравнения.

3. Большинство клиентов имеют один профиль, но зона дублей всё равно существенная: 15.13% клиентов имеют несколько профилей, а 26.86% профилей относятся к таким клиентам.

4. Классические identity-поля разрежены: телефон есть только у 7.52% профилей, фамилия и дата рождения заполнены очень слабо. Поэтому нельзя строить поиск дублей только на ФИО, телефоне и дате рождения.

5. Основной рабочий сигнал находится в feature-колонках: `realtime_features`, `fs_features`, `non_processing_features` заполнены почти на 99.6% строк и дальше должны быть разобраны в multi-value витрину.

6. `entity_id` используется только как ground truth для обучения и оценки качества. В production-признаки его включать нельзя.
